# Code Convention

이 문서는 Log Analyzer 프로젝트에서 사용하는 **코드 설계 원칙과 작성 기준**을 설명한다.
본 프로젝트의 모든 PR은 이 문서를 기준으로 리뷰된다.

---

## 1. 객체 생성과 수명 관리

본 프로젝트는 요청 단위로 다수의 객체를 생성한다.  
기본 생성자 호출을 그대로 노출하기보다는, 객체가 어떤 의미로 만들어졌는지를 코드에서 드러내는 방식을 선호한다.
또한 객체를 필요 이상으로 오래 참조하지 않는 것을 중요한 설계 원칙으로 둔다.

불필요한 객체 생성을 피하기 위해,
- 값 객체는 재사용 가능하면 정적 팩터리로 제공한다.
- 동일한 의미의 객체를 반복 생성하지 않도록 한다.

또한 다 쓴 객체에 대한 참조를 명시적으로 끊는다.  
요청 처리 이후에도 유지되어야 할 이유가 없는 객체는  
컬렉션이나 필드에 보관하지 않는다.

이 원칙은 가비지 컬렉션의 부담을 줄이고,  
대량 로그 처리 시 메모리 사용량이 급증하는 상황을 방지한다.

### 정적 팩터리 메서드 사용

객체 생성에는 가능하면 생성자 대신 정적 팩터리 메서드를 사용한다.  
이는 생성 시점의 **의도를 명확히 표현**하고, 매개변수 검증과 기본값 처리 로직을 한 곳에 모으기 위함이다.

```java
IpInfo.unknown(ip);
IpInfo.of(ip, country, region, city, organization);
```

위와 같은 호출은 단순히 객체를 만든다는 사실보다,
“이 IP는 정보가 없는 상태다”, “이 IP는 검증된 정보로 생성된다”라는 의미를 함께 전달한다.

또한 향후 캐싱, 구현 교체, 생성 로직 변경이 필요해질 경우에도
호출부 변경 없이 내부 구현만 수정할 수 있다는 장점이 있다.

### Builder 패턴 사용

필드가 많거나 선택적 값이 존재하는 객체의 경우 Builder 패턴을 사용한다.
이는 가독성을 높이고, 잘못된 매개변수 순서로 인한 오류를 방지하기 위함이다.

```java
AnalysisResult result = AnalysisResult.builder()
    .analysisId(id)
    .completedAt(now)
    .statistics(statistics)
    .build();
```
Builder를 사용함으로써 객체는 생성 이후 상태가 변하지 않는 불변 객체로 유지된다.

### 의존 객체 주입

객체가 사용하는 다른 객체들은 내부에서 직접 생성하지 않고, 외부에서 주입받도록 설계한다.

이 방식은 테스트 시 Mock 객체를 주입할 수 있게 하며,
객체의 책임과 역할을 명확히 분리해 준다.

Spring 환경에서는 생성자 주입을 기본 원칙으로 사용한다.

### 자원 관리

파일, 스트림 등 명시적인 자원 해제가 필요한 경우에는
try-with-resources 구문을 사용한다.

이는 예외 발생 여부와 관계없이 자원이 안전하게 해제되도록 보장하며,
자원 누수로 인한 장애 가능성을 줄인다.

## 2. 불변성과 스레드 안정성

본 프로젝트의 도메인 모델은 기본적으로 불변 객체를 전제로 설계한다.

### 불변 도메인 객체

도메인 객체는 Java 17의 record 또는 Lombok의 `@Value`를 사용하여 정의한다.
객체는 생성 이후 상태가 변경되지 않으며, 모든 변경은 새 객체를 생성하는 방식으로 처리한다.

```java
public record AccessLog(
    LocalDateTime timeGenerated,
    String clientIp,
    int httpStatus
) {}
```

불변 객체는 별도의 동기화 없이도 스레드 안전하며,
객체의 상태를 추론하기 쉬워 디버깅과 유지보수에 유리하다.

### 값 객체와 동등성 규약


도메인 모델은 값 객체(Value Object)의 성격을 가진다.  
따라서 동등성은 참조가 아니라 값 기준으로 판단한다.

Java record를 사용함으로써  
equals, hashCode, toString의 일반 규약을 자동으로 만족시키며,  
직접 재정의로 인한 오류 가능성을 제거한다.

이는 컬렉션(Map, Set) 사용 시에도 예상 가능한 동작을 보장하는 중요한 전제다.

### 내부 상태 보호

객체 내부에서 컬렉션을 보관하는 경우,
외부로 그대로 노출하지 않는다.

생성자에서는 List.copyOf, Map.copyOf 등을 사용해 방어적 복사를 수행하고,
반환 시에는 수정 불가능한 뷰 또는 복사본을 반환한다.

이를 통해 외부 코드가 객체의 내부 상태를 의도치 않게 변경하는 것을 방지한다.

### 동시성 자료구조

여러 요청이 동시에 접근할 수 있는 저장소 성격의 컴포넌트에서는
동기화된 컬렉션 또는 ConcurrentHashMap과 같은 thread-safe 자료구조를 사용한다.

동시성 제어는 명시적으로 드러나야 하며, 암묵적인 동기화에 의존하지 않는다.

## 3. 메서드 계약과 방어적 프로그래밍

모든 public 메서드는 명확한 입력 조건과 반환 규칙을 가져야 한다.

### 매개변수 유효성 검사

메서드는 잘못된 입력을 조용히 허용하지 않는다.
null이 허용되지 않는 경우, 메서드 진입 시점에서 즉시 검증한다.

```java
Objects.requireNonNull(file, "file은 null일 수 없습니다");
```
이러한 Fail-Fast 전략은 오류를 조기에 발견하게 해주며,
문제의 원인을 추적하기 쉽게 만든다.

### null 반환 금지

메서드는 null을 반환하지 않는다.
값이 없음을 표현해야 하는 경우 Optional을 사용하거나,
컬렉션의 경우 빈 컬렉션을 반환한다.

```java
return Collections.emptyList();
```
이 규칙은 호출부에서 불필요한 null 체크를 제거하고,
API 사용 방식을 일관되게 만든다.

### 방어적 복사
외부에서 전달받은 컬렉션이나 배열은 그대로 저장하지 않는다.
생성자 또는 메서드 내부에서 복사본을 만들어 사용한다.

이는 외부 변경이 내부 상태에 영향을 주는 것을 차단하기 위함.

## 4. 계층 분리 원칙

프로젝트는 다음과 같은 계층 구조를 따른다.

Domain 레이어는 순수한 비즈니스 개념만 표현하며,
Spring, HTTP, DB, 외부 API와 같은 기술적 요소에 의존하지 않는다.

Application 레이어는 도메인 객체를 조합하여 유스케이스를 수행하는 역할을 맡는다.

Infrastructure 레이어는 외부 시스템과의 연동 책임을 가지며,
기술 변경 가능성을 이 레이어 안으로 제한한다.

이러한 분리는 코드 변경의 영향을 최소화하고, 테스트 범위를 명확히 한다.

## 접근 제어와 API 경계

클래스와 멤버의 접근 범위는 가능한 한 최소화한다.  
외부에서 사용할 필요가 없는 타입이나 메서드는 public으로 노출하지 않는다.

public API는 곧 계약이다.  
한 번 공개된 API는 변경 비용이 크기 때문에, 외부 노출은 신중하게 결정한다.

이를 통해 내부 구현 변경이 외부 코드에 영향을 주지 않도록 한다.

## 5. 예외와 로깅

예외 메시지는 무엇이 잘못되었는지를 명확히 설명해야 한다.
기술적인 예외를 그대로 외부에 노출하지 않으며,
의미 있는 도메인 또는 API 수준의 메시지로 변환한다.

로그는 성공과 실패를 구분하여 남기며,
토큰이나 개인정보와 같은 민감 정보는 절대 기록하지 않는다.

## 정리

이 프로젝트의 코드 컨벤션은 다음 원칙을 중심으로 설계되었다.

1. 객체는 명확한 의미를 가지고 생성되며, 상태는 가능한 한 불변으로 유지된다.

2. 메서드는 명확한 계약을 가지며, null과 예측 불가능한 동작을 허용하지 않는다.

3. 계층은 명확히 분리되고, 기술적 변화는 외부로 격리된다.
